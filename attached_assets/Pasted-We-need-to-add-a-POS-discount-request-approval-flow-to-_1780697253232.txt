We need to add a POS discount request approval flow to Gaming Lounge OS without breaking any existing functionality.

Main goal:
Cashier users should NOT be able to apply discounts directly.
Cashiers can only request a discount percentage from the POS.
The owner can approve or reject the discount request.
If approved, the discount is applied to the current POS order/session bill.
If rejected, the discount is not applied.

Important:
- Do not break existing POS, orders, payments, sessions, shifts, dashboard, inventory, finance, auth, tenant system, or permissions.
- This must be multi-tenant and fully scoped by tenant_id.
- Add this feature safely and additively.
- Do not change existing payment verification logic.
- Do not allow unauthorized users to approve discounts.

Business logic:
1. Cashier opens POS.
2. Cashier creates or edits an order.
3. Cashier clicks “Request Discount”.
4. Cashier enters discount percentage and optional reason.
5. The request is sent to the owner/manager approval queue.
6. The order stays without discount until approval.
7. Owner sees the discount request in POS or notifications.
8. Owner can Approve or Reject.
9. If approved:
   - Apply the discount to the target order/session bill.
   - Recalculate subtotal, discount amount, tax if applicable, and final total.
   - Save who approved it and when.
10. If rejected:
   - Do not apply any discount.
   - Save who rejected it and when.
   - Show cashier that the request was rejected.

Discount rules:
- Discount must be percentage-based.
- Minimum discount: 0%
- Maximum discount: configurable, but default max should be 50%.
- Cashier cannot approve their own request.
- Cashier cannot edit a discount after it is approved.
- Owner can approve or reject.
- Platform Owner can approve or reject.
- Manager can approve only if permission is granted.
- A discount request can have only one final status: pending, approved, rejected, cancelled.
- Only one active pending discount request should exist per order/session bill at a time.
- If the order is already paid, do not allow a new discount request.
- If the bill is closed/paid, approved discounts should not change past verified payments unless a clear refund/adjustment flow already exists.

Database:
Add a new table called discount_requests.

Fields:
- id
- tenant_id
- order_id nullable
- session_id nullable
- payment_id nullable
- requested_by user id
- approved_by user id nullable
- discount_percent numeric
- discount_amount numeric nullable
- original_total numeric
- final_total numeric nullable
- reason text nullable
- owner_note text nullable
- status: pending, approved, rejected, cancelled
- requested_at
- reviewed_at nullable
- created_at
- updated_at

Important:
- Either order_id or session_id should be linked depending on where the discount is requested.
- Keep fields nullable where needed to avoid breaking existing records.
- Do not modify existing tables unless absolutely required.
- If existing orders already have discount fields, reuse them safely.
- If not, add optional nullable fields only:
  - discount_percent
  - discount_amount
  - discount_request_id
  - discount_approved_by
  - discount_approved_at

Backend API:
Create tenant-scoped secure endpoints:

- POST /api/pos/discount-requests
  Cashier creates a discount request.

Payload:
- order_id or session_id
- discount_percent
- reason

Validation:
- user must have permission discount.request
- order/session must belong to same tenant
- order/session must not be paid/closed
- discount_percent must be valid
- no duplicate pending request for same target

- GET /api/pos/discount-requests
  Owner/manager gets pending/recent requests.

Query filters:
- status
- date range
- requested_by
- order_id
- session_id

- POST /api/pos/discount-requests/:id/approve
  Owner approves request.

Payload:
- optional owner_note

Behavior:
- user must have permission discount.approve
- request must be pending
- apply discount to the linked order/session bill
- recalculate totals
- save approved_by and reviewed_at
- status becomes approved
- cashier sees updated bill in POS

- POST /api/pos/discount-requests/:id/reject
  Owner rejects request.

Payload:
- optional owner_note

Behavior:
- user must have permission discount.approve
- request must be pending
- status becomes rejected
- do not change order/session total
- cashier sees rejection message

- POST /api/pos/discount-requests/:id/cancel
  Cashier can cancel their own pending request before review.

Discount request = 10%

final_total = original_total - discount_amount
